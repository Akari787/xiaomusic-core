# v1.0.10 发布说明

发布日期：2026-04-09  
适用分支：`main`

---

## 1. 核心变更摘要

v1.0.10 不是功能扩张版本，这一版的目标是：

- 把 auth 文档主线与当前真实实现对齐
- 把 `_try_login()` 主路径的阶段边界写清楚
- 把 playback 起播确认的局部验收结论收口到正式发布口径
- 把“已通过的是哪一层，不是哪一层”固定到文档与发布说明里

这一版的重点不是“新增恢复策略”，而是**把当前真实主线、验收边界和发布口径统一起来**。

---

## 2. auth 主线的真实变化

当前 auth 主线已经更新为：

1. `_try_login()` 作为当前 auth runtime 恢复核心路径之一
2. 恢复登录前使用 fresh `ClientSession`
3. `MiAccount.login("micoapi")` 成功后，才允许构建 candidate runtime
4. candidate runtime 的 account / cookie ready 后，才允许进入 verify
5. verify 成功后，才允许 runtime swap
6. login-stage failure 与 verify-stage failure 明确区分
7. verify 失败时，候选 runtime 不得污染旧 runtime

本版同时完成了 auth 文档主线与现实现状的对齐，不再把旧的“自动登录 fallback 被禁用”表述当作当前主线。

---

## 3. playback 起播确认收口

本版没有新增播放能力，但把当前 playback 的局部通过范围写清楚了：

- `local_library` 首轮 start confirmation 有局部实机证据
- `direct_url` 首轮 start confirmation 有局部实机证据
- player state 契约测试仍作为状态面约束基础

这代表：

- 当前发布可以确认“起播确认机制已有局部证据”
- 不能扩写成“playback 全量长期稳定已完成验收”

## 5. 本次发布已确认通过的范围

### 5.1 auth

本次发布确认的是：

- **fresh session 修补后的 `_try_login()` 主路径稳定性**
- 在本轮观察内，`/api/auth/status` 与 debug 业务结论一致并指向 healthy
- login-stage failure 与 verify-stage failure 已有明确分层口径

### 5.2 playback

本次发布确认的是：

- `local_library` 首轮 start confirmation 已有局部实机证据
- `direct_url` 首轮 start confirmation 已有局部实机证据

### 5.3 文档与发布口径

本次发布确认的是：

- auth 主线文档已与当前真实实现对齐
- 发布文档、导航入口、验收矩阵与 checklist 已形成统一口径

---

## 6. 本次发布未覆盖范围 / 已知边界

本次发布**没有确认**以下内容：

- spec rebuild 全量通过
- auto runtime reload 全量通过
- singleflight 全量实机闭环
- fallback path 全量独立验收
- auth / playback 交叉边界
- 极端网络扰动恢复行为
- playback 长时间无播放 / 持续播放 / 混合负载的全量长期行为

已知边界仍然包括：

- Xiaomi 云端风控仍不可消除
- 当前目标仍是可恢复、可观测、可预测，不是零失败
- 当前主路径通过，不等于全部恢复分支通过

---

## 7. 本次发布确认的是哪一层，不是哪一层

### 本次发布确认的是

- auth runtime 恢复主路径中 **fresh session `_try_login()`** 这一层
- status/debug 在本轮观察内的业务一致性这一层
- playback 首轮 start confirmation 的局部 smoke 这一层
- 文档主线与发布口径收口这一层

### 本次发布没有确认的不是

- spec rebuild 全量
- auto runtime reload 全边界
- singleflight / fallback / cross-boundary 全量
- 全系统长期稳定性总验收
