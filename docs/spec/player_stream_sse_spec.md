# 播放器状态 SSE 推送协议规范（Player Stream SSE Spec）

版本：v1.0
状态：正式规范
最后更新：2026-03-22
适用范围：`GET /api/v1/player/stream`、WebUI SSE 消费路径、服务端 SSE 推送实现

---

## 1. 文档定位

本文档定义 `GET /api/v1/player/stream` 的 SSE（Server-Sent Events）推送协议，包括接口定义、连接行为、事件结构、推送时机、重连机制与前端消费约束。

本文档是**传输协议规范**，不重新定义播放状态字段的语义。SSE 事件中携带的状态数据，必须完全遵守 `docs/spec/player_state_projection_spec.md`（以下简称"投影规范"）所定义的权威播放状态快照模型。本文档不引入任何投影规范之外的字段语义。

### 1.1 SSE 是播放状态的主通道

`GET /api/v1/player/stream` 是前端获取播放器状态的**主通道**。在 SSE 连接正常建立且运行的情况下，前端必须通过 SSE 接收播放状态更新，而不是通过轮询。

`GET /api/v1/player/state` 的定位为**初始化与降级回退通道**，适用于以下场景：

- 页面首次加载、SSE 连接尚未建立时，用于获取初始状态快照。
- SSE 连接断开且重连尚未成功时，用于轮询兜底。
- 调试与状态核验目的。

任何情况下，前端都不得将 SSE 与 `/player/state` 轮询并行运行并让两者竞争写入 UI。在 SSE 连接正常运行期间，轮询必须停止。

### 1.2 废弃说明

本文档所定义的 SSE 机制取代了历史上曾讨论或试用过的 WebSocket 状态推送方案。WebSocket 方案不属于本规范的范围，不得在新实现中引入。

## 2. 接口定义

### 2.1 路径与方法

```
GET /api/v1/player/stream
```

本接口属于 v1 Public API，遵守 `docs/api/api_v1_spec.md` 的命名空间与分级规则。本接口归属 Class C（查询型），不要求 `transport`，不触发任何设备侧动作。

### 2.2 查询参数

| 参数 | 类型 | 必须 | 说明 |
|---|---|---|---|
| `device_id` | `string` | 是 | 订阅的目标设备 ID，非空字符串 |

`device_id` 必须是已注册设备的合法 ID。一个 SSE 连接只订阅一个设备，不允许省略 `device_id` 或通过通配符订阅多个设备。

### 2.3 请求 Header

客户端发起请求时，应携带以下 Header：

| Header | 值 | 说明 |
|---|---|---|
| `Accept` | `text/event-stream` | 表明客户端期望接收 SSE 流 |
| `Cache-Control` | `no-cache` | 禁止中间缓存干扰 SSE 流 |

### 2.4 响应 Header

服务端响应必须包含以下 Header：

| Header | 值 |
|---|---|
| `Content-Type` | `text/event-stream; charset=utf-8` |
| `Cache-Control` | `no-cache` |
| `Connection` | `keep-alive` |
| `X-Accel-Buffering` | `no` |

`X-Accel-Buffering: no` 必须存在，用于禁止 Nginx 等反向代理对 SSE 流进行缓冲，确保事件能实时到达客户端。

### 2.5 HTTP 状态码

| 状态码 | 语义 |
|---|---|
| `200 OK` | 连接建立成功，SSE 流开始 |
| `400 Bad Request` | 缺少 `device_id` 或参数非法 |
| `401 Unauthorized` | 鉴权失败 |
| `404 Not Found` | 指定 `device_id` 对应的设备不存在 |
| `503 Service Unavailable` | 服务端当前无法提供 SSE 服务 |

`400`、`401`、`404`、`503` 响应必须遵守 `api_v1_spec.md` 的统一错误模型，以 JSON 格式返回结构化错误（`code`、`message`、`data.error_code`、`data.stage`、`request_id`），而不是纯文本错误消息。

## 3. 连接行为

### 3.1 建立连接后的初始推送

服务端在 SSE 连接成功建立后，必须**立即**推送一条 `player_state` 事件，携带该设备当前最新的完整状态快照。

此初始推送是强制行为，不得省略。其目的是确保前端在切换到 SSE 模式时，能够立即得到一份权威状态，而不是等待下一次状态变化才收到第一条事件。

### 3.2 长连接语义

本接口是长连接，连接建立后应持续保持，直至以下情况之一发生：

- 客户端主动关闭连接（例如页面卸载、用户导航离开）。
- 服务端发生无法恢复的异常，需要主动关闭连接（见第 3.3 节）。

在连接存续期间，服务端不得因为播放器状态长时间未变化而主动关闭连接。长时间无状态变化时，服务端必须依靠心跳机制（见第 8 节）维持连接活跃，而不是关闭连接。

### 3.3 服务端主动关闭的限制

服务端仅在以下情况下允许主动关闭 SSE 连接：

- 服务端进程即将退出或重启。
- 客户端的会话鉴权已过期且无法自动续期（服务端在关闭前应推送一条错误事件，见第 10.3 节）。
- 服务端内部出现无法恢复的错误，导致无法继续推送状态。

服务端不得因以下原因主动关闭连接：

- 播放器进入 `idle` 或 `stopped` 状态。
- 订阅的设备当前不在播放中。
- 长时间未产生新的播放事件（此时应使用心跳维持）。

## 4. 事件结构

### 4.1 标准 SSE 格式

所有播放状态事件必须使用以下标准 SSE 格式：

```
event: player_state
id: <event_id>
data: <JSON 序列化的状态快照>

```

每条事件以空行结束（即 `\n\n`），这是标准 SSE 协议要求。

### 4.2 字段说明

**`event`**：事件类型，播放状态事件的类型固定为 `player_state`，不得使用其他值。

**`id`**：事件 ID，必须存在。其值必须为该事件所携带状态快照的 `revision` 值（字符串形式）。此设计使浏览器在自动重连时能通过 `Last-Event-ID` 请求头上报最后收到的事件 ID，服务端可据此判断客户端状态（见第 7 节）。

**`data`**：必须是合法 JSON，内容必须是完整的权威播放状态快照对象，完全符合投影规范第 5 节所定义的字段模型。不允许对快照字段做任何裁剪、重命名或结构变形。

### 4.3 data 字段的完整性要求

`data` 字段中的状态快照必须包含投影规范中所有标记为"必须存在"的字段，具体包括：

- `device_id`
- `revision`
- `play_session_id`
- `transport_state`
- `track`（对象或 `null`，含 `track.id`、`track.title`）
- `context`（对象或 `null`，含 `context.id`、`context.name`、`context.current_index`）
- `position_ms`
- `duration_ms`
- `snapshot_at_ms`

每次推送必须携带完整快照，不得仅推送变化的字段（即不使用增量/差异推送）。前端必须每次以完整快照替换本地状态，而不是将事件数据合并到旧状态上。

### 4.4 心跳事件格式

心跳事件不是 `player_state` 事件，必须使用 SSE 注释格式，不携带 `event` 或 `data` 字段：

```
: heartbeat

```

心跳事件不携带状态数据，前端收到心跳后仅需重置连接超时计时器，不得将其作为状态更新处理。

### 4.5 错误事件格式

当服务端需要在关闭连接前通知客户端发生了错误（见第 10.3 节），应推送一条 `stream_error` 事件，然后关闭连接：

```
event: stream_error
data: {"error_code": "<ERROR_CODE>", "message": "<说明>"}

```

`stream_error` 事件中的 `error_code` 应使用 `api_v1_spec.md` 中定义的错误码体系，`message` 为可读说明。客户端收到 `stream_error` 后，应根据 `error_code` 决定是否发起重连，而不是无条件重连。

## 5. 推送时机

### 5.1 必须推送新事件的情况

以下任何情况发生时，服务端必须推送一条新的 `player_state` 事件。这些情况与投影规范第 6.2 节定义的 `revision` 递增触发条件完全对应：

- **曲目切换**：当前播放曲目切换到另一首，`play_session_id` 发生变化。
- **传输状态变化**：`transport_state` 枚举值发生任何变化，包括但不限于 `playing → switching`、`switching → playing`、`playing → paused`、`paused → playing`、`playing → stopped`、任意状态 → `error`。
- **播放会话变化**：`play_session_id` 发生变化的任何场景。
- **曲目身份变化**：`track.id` 发生变化。
- **上下文变化**：`context.id` 或 `context.current_index` 发生变化。
- **元数据延迟到达**：在 `track.id` 保持稳定的情况下，`track.title`、`duration_ms` 等 metadata 字段从空值或占位值更新为正式值，应推送一次更新事件（此时 `revision` 应递增）。
- **连接建立**：任何新连接建立后，服务端必须立即推送一次完整的当前快照（见第 3.1 节）。

### 5.2 不得推送新事件的情况

以下情况不得触发新的 `player_state` 事件推送：

- **进度的自然推进**：播放过程中 `position_ms` 因时间流逝而增加。前端通过本地插值处理进度推进（见投影规范第 10.3 节），服务端不为此推送事件，也不递增 `revision`。
- **内部状态变化但对外快照不变**：服务端内部缓存刷新、日志记录、非播放相关配置变化等，不影响投影规范定义的任何快照字段时，不得推送事件。
- **重复状态**：若服务端内部产生了与上一次推送快照完全相同（`revision` 相同）的快照，不得再次推送。

### 5.3 推送顺序保证

服务端应确保在同一 SSE 连接上，事件的推送顺序与 `revision` 的递增顺序一致。若因内部并发导致顺序可能乱序，服务端在推送前必须先完成排序，而不是依赖前端对乱序做容错处理。

前端的 `revision` 去重逻辑（见第 6 节）是针对重连、慢速网络等传输层因素的防御机制，不是弥补服务端推送乱序的手段。

## 6. revision 与去重

### 6.1 revision 的传输角色

`revision` 在 SSE 传输协议中承担以下角色：

- 通过 SSE `id` 字段传递，使浏览器重连时能携带最后已知的 revision。
- 作为前端去重的唯一依据，不依赖事件的到达顺序。

### 6.2 前端去重规则

前端在收到每一条 `player_state` 事件后，必须执行以下判断：

1. 解析 `data` 字段，读取 `revision` 值。
2. 将收到的 `revision` 与当前 UI 已渲染状态的 `revision` 进行比较。
3. 若收到的 `revision` 严格小于当前已渲染的 `revision`，必须丢弃该事件，不得更新 UI。
4. 若收到的 `revision` 等于当前已渲染的 `revision`，视为重复事件，应丢弃，不触发 UI 重渲染。
5. 若收到的 `revision` 大于当前已渲染的 `revision`，应用该快照更新 UI。

### 6.3 服务端不得依赖顺序假设

服务端在设计推送逻辑时，不得假设"事件一定按顺序到达前端"。以下场景可能导致事件乱序或重复到达：

- 网络拥塞导致事件延迟，与重连后的新事件交错到达。
- 客户端重连后收到初始快照，而此前发送的旧事件因网络缓冲延迟也在此时到达。
- 代理或中间件对 SSE 流进行缓冲后批量转发。

服务端的职责是确保每条事件的 `revision` 正确，前端的职责是基于 `revision` 做去重，双方不得依赖传输层顺序保证。

## 7. 重连机制

### 7.1 浏览器自动重连行为

浏览器的 `EventSource` API 在连接断开后会自动发起重连，默认重连间隔约为 3 秒（具体行为由浏览器实现决定）。重连时，浏览器会在请求中携带 `Last-Event-ID` Header，其值为最后收到的 SSE `id` 字段值（即最后已知的 `revision`）。

服务端应接受携带 `Last-Event-ID` 的重连请求，并识别其中的值，但不得依赖 `Last-Event-ID` 实现增量补发逻辑（见第 7.2 节）。

### 7.2 重连后必须推送完整快照

无论客户端重连时携带了何种 `Last-Event-ID`，服务端在重连连接建立后，必须立即推送当前设备的完整最新状态快照（full snapshot），而不是从客户端上报的 `revision` 之后开始补发历史事件。

理由：服务端不得假设客户端在断线期间的内部状态是完整且未被篡改的；完整快照是最安全、最简洁的重新同步方式。服务端不维护历史事件队列，也不做增量补发。

### 7.3 重连间隔控制

服务端可以通过 SSE `retry` 字段建议客户端的重连间隔（单位：毫秒）。建议值为 `3000`（3 秒）至 `10000`（10 秒）之间：

```
retry: 5000

```

此字段应在连接建立时随初始快照事件一起发送，不需要在每条事件中重复。

### 7.4 重连期间的降级行为

在 SSE 连接断开、重连尚未成功的期间，前端应启动对 `GET /api/v1/player/state` 的轮询作为降级兜底（见第 11 节）。一旦 SSE 重连成功并收到初始快照，前端必须立即停止轮询，切回 SSE 主通道。

## 8. 心跳机制

### 8.1 心跳是必须实现的功能

服务端必须定期向所有活跃的 SSE 连接发送心跳，以防止以下问题：

- 客户端或中间代理因长时间无数据而错误地判断连接已断开，触发不必要的重连。
- Nginx、负载均衡器等组件因空闲超时主动关闭 TCP 连接。

### 8.2 心跳格式

心跳必须使用 SSE 注释格式（见第 4.4 节），不得伪装为 `player_state` 事件：

```
: heartbeat

```

心跳注释行以冒号开头，符合 SSE 规范中注释行的定义，不会被浏览器 `EventSource` 的 `onmessage` 回调接收，不会触发任何事件处理逻辑。

### 8.3 心跳频率

心跳发送间隔建议为 **15 秒**，不得超过 **30 秒**。具体间隔可由实现侧根据部署环境（如反向代理的空闲超时配置）调整，但必须有心跳机制，不允许不发送任何心跳。

若距上次推送 `player_state` 事件的时间已超过心跳间隔，服务端必须发送心跳，无论其间是否有播放状态变化。

## 9. 多设备场景

### 9.1 一个连接只订阅一个设备

每个 SSE 连接必须绑定到单个 `device_id`，只推送该设备的状态事件。服务端不得在同一连接上混合推送多个设备的状态。

### 9.2 device_id 的过滤约束

服务端在推送事件时，必须严格过滤，确保每条推送的 `player_state` 事件中的 `device_id` 字段，与建立连接时请求参数中的 `device_id` 完全一致。如果内部状态更新来自其他设备，不得通过当前连接推送。

### 9.3 多设备监控

若前端需要同时监控多个设备的播放状态，必须为每个设备建立独立的 SSE 连接，分别携带各自的 `device_id` 参数。服务端不提供多设备聚合推送能力，前端不得向单个连接传递多个 `device_id`。

## 10. 错误处理

### 10.1 参数错误（400）

若请求中缺少 `device_id` 或 `device_id` 为空字符串，服务端必须拒绝请求，返回 HTTP 400，不建立 SSE 连接。响应体必须符合 `api_v1_spec.md` 的统一 envelope 格式：

```json
{
  "code": 40001,
  "message": "missing or invalid device_id",
  "data": {
    "error_code": "E_INVALID_REQUEST",
    "stage": "request"
  },
  "request_id": "req_xxx"
}
```

### 10.2 设备不存在（404）

若 `device_id` 对应的设备在当前运行时中不存在，服务端必须返回 HTTP 404，不建立 SSE 连接。

```json
{
  "code": 40004,
  "message": "device not found",
  "data": {
    "error_code": "E_DEVICE_NOT_FOUND",
    "stage": "xiaomi"
  },
  "request_id": "req_xxx"
}
```

### 10.3 连接建立后的运行时错误

若 SSE 连接已建立（HTTP 200 已发出），但服务端随后遇到需要关闭连接的运行时错误（例如鉴权过期、设备注销），服务端应在关闭连接前推送一条 `stream_error` 事件（格式见第 4.5 节），然后关闭连接。

客户端收到 `stream_error` 事件后，应根据 `error_code` 决定后续行为：

| `error_code` | 建议行为 |
|---|---|
| `E_AUTH_EXPIRED` | 触发认证刷新流程，刷新成功后重新建立 SSE 连接 |
| `E_DEVICE_NOT_FOUND` | 提示用户设备已离线，停止重连 |
| `E_SERVER_SHUTDOWN` | 等待服务恢复后重连，建议使用指数退避 |

对于未在上表列出的 `error_code`，客户端应使用默认重连策略（指数退避，最大间隔不超过 60 秒）。

### 10.4 鉴权失败（401）

若项目启用了 API 鉴权，`GET /api/v1/player/stream` 请求在鉴权校验阶段失败时，必须返回 HTTP 401，不建立 SSE 连接。

鉴权校验机制与项目现有 Internal API 鉴权体系一致，具体由 `docs/spec/auth_runtime_recovery.md` 约束，本文档不重复定义鉴权细节。

### 10.5 服务不可用（503）

若服务端当前因进程启动中、重载中或其他原因无法提供 SSE 服务，应返回 HTTP 503。客户端应在收到 503 后使用指数退避策略进行重连，初始等待时间不低于 5 秒。

## 11. 与 HTTP player/state 接口的关系

### 11.1 定位分工

| 接口 | 定位 | 适用场景 |
|---|---|---|
| `GET /api/v1/player/stream` | **主通道** | SSE 连接正常时的所有状态接收 |
| `GET /api/v1/player/state` | **初始化 + 降级回退** | 页面首次加载、SSE 断线重连期间的轮询兜底、调试核验 |

### 11.2 两者基于同一快照构建器

`/player/stream` 推送的 `player_state` 事件数据，与 `/player/state` 响应中 `data` 的内容，必须由同一个状态快照构建器生成（见投影规范第 11 节）。两个接口不得维护各自独立的状态拼装逻辑。

同一时刻，若 `/player/state` 返回的快照与 `/player/stream` 最近推送的快照存在差异，以 `revision` 较大的为准。若两者 `revision` 相同但字段内容不同，应判定为快照构建器实现存在缺陷，需修复实现，而不是在文档层面兼容差异。

### 11.3 禁止并行竞争

在 SSE 连接正常运行期间，前端不得同时对 `/player/state` 发起轮询。两路状态来源并行运行，将导致不同 `revision` 的快照交替写入 UI，产生状态竞争与视觉闪烁。

以下行为明确禁止：

- SSE 连接正常时，仍以固定间隔轮询 `/player/state` 并将结果合并到 UI。
- 以"SSE 可能不及时"为由，在 SSE 之外保留一条并行的状态轮询链路。
- 前端将 SSE 事件与 HTTP 轮询结果合并后，自行裁决哪一份"更准确"。

## 12. 前端消费约束

### 12.1 SSE 为主通道，轮询为降级

前端必须优先建立 SSE 连接作为播放状态的主获取通道。页面加载时的初始化流程应为：

1. 通过 `GET /api/v1/player/state` 获取初始快照，立即渲染初始状态。
2. 建立 `GET /api/v1/player/stream` 的 SSE 连接。
3. SSE 连接建立并收到初始 `player_state` 事件后，停止任何轮询，以 SSE 为唯一状态来源。

若 SSE 连接建立失败（非 200 响应）或断开后重连失败，前端应切换为对 `/player/state` 的轮询模式，轮询间隔不低于 3 秒。一旦 SSE 重连成功并收到初始快照，必须立即停止轮询。

### 12.2 revision 去重必须执行

前端在处理每一条 `player_state` 事件时，必须执行第 6.2 节定义的 `revision` 去重逻辑，不得因"SSE 通常有序"而跳过去重检查。

### 12.3 完整快照替换，禁止增量合并

每次收到 `player_state` 事件时，前端必须以事件中的完整快照替换本地状态，不得将事件 `data` 字段与已有状态做合并（merge）操作。增量合并逻辑可能导致旧字段残留，产生状态污染。

### 12.4 play_session_id 与 track.id 绑定行为

前端在处理 SSE 事件时，关于 `play_session_id`、`track.id` 的消费规则与投影规范第 12.3 节完全一致，本文档不重复定义，应以投影规范为准。

### 12.5 心跳事件的处理

前端不得将心跳注释（`: heartbeat`）视为有效事件处理。浏览器 `EventSource` API 会自动忽略注释行，前端代码不需要为此添加额外处理逻辑。若使用自定义 SSE 解析器，必须确保其正确忽略以冒号开头的注释行。

### 12.6 stream_error 事件的处理

前端必须监听 `stream_error` 事件类型。收到后，不得无条件立即重连，应先读取 `error_code`，按第 10.3 节的建议行为处理。特别地，收到 `E_AUTH_EXPIRED` 时必须先完成认证刷新，再重新建立 SSE 连接；未完成认证刷新前，不得循环重连。

### 12.7 页面可见性变化

当页面因切换到后台（`document.visibilityState == "hidden"`）导致 SSE 连接可能被浏览器暂停或断开时，前端应在页面重新变为可见（`visibilitychange` 事件触发且 `document.visibilityState == "visible"`）时检查 SSE 连接状态，并在连接已断开的情况下主动发起重连，而不是等待浏览器自动重连超时。

## 13. 与其他规范的关系

### 13.1 与投影规范的关系

`docs/spec/player_state_projection_spec.md` 是本文档的上位规范。本文档定义的 SSE 传输协议，负责将投影规范所定义的状态快照通过 SSE 通道推送给消费方。本文档不定义快照的字段语义，所有字段语义以投影规范为准。

### 13.2 与 API v1 契约的关系

`docs/api/api_v1_spec.md` 是 v1 Public API 的唯一权威契约来源。`GET /api/v1/player/stream` 作为 v1 白名单接口，须遵守 `api_v1_spec.md` 的通用协议要求，包括命名空间约束、统一错误模型（适用于非 200 错误响应场景）与接口分级规则（Class C）。

### 13.3 与 WebUI 相关规范的关系

`docs/spec/webui_playback_state_machine_spec.md` 定义 WebUI 如何消费播放状态并维护展示层状态机。WebUI 状态机的输入来源，在 SSE 可用时必须是本文档定义的 SSE 事件；在 SSE 不可用时才降级为 `/player/state` 轮询。WebUI 规范中凡涉及"以哪个字段为权威依据"的规则，以投影规范为准，不受传输通道变化影响。

### 13.4 与 PlaybackCoordinator 的关系

PlaybackCoordinator 的内部状态变化是服务端产生推送事件的来源。当 PlaybackCoordinator 产生导致 `revision` 递增的状态变化时，必须通知 SSE 推送组件，由其通过统一快照构建器生成快照并推送。PlaybackCoordinator 不直接写入 SSE 流；SSE 推送组件通过快照构建器与 PlaybackCoordinator 解耦。
