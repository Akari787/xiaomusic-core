# 播放器权威状态快照规范（Player State Projection Spec）

版本：v1.0
状态：正式规范
最后更新：2026-03-22
适用范围：`GET /api/v1/player/state`、未来的 `GET /api/v1/player/stream`（SSE）、PlaybackCoordinator / PlaybackFacade、WebUI 状态消费

---

## 1. 文档定位

本文档定义 XiaoMusic 播放器"权威播放状态快照"（Player State Projection）的结构、字段语义与消费约束。

本文档是以下内容的共同上位规范：

- `GET /api/v1/player/state` 的响应数据结构与语义
- 未来 `GET /api/v1/player/stream` SSE 事件中 `data` 的状态结构与语义
- PlaybackCoordinator 与 PlaybackFacade 对外暴露状态的约束边界
- WebUI、Home Assistant、第三方调用方消费播放状态的合法边界

本文档做出以下强制约束：

1. **所有对外状态输出必须共享同一套状态投影逻辑。** HTTP 状态查询与 SSE 状态推送不得分别维护独立的状态拼装逻辑；两者必须基于同一个状态快照构建器生成。
2. **前端不得以本地启发式逻辑（heuristic）或轮询补偿逻辑作为播放真相来源。** 服务端投影是唯一权威来源，前端是消费者，不是裁决者。
3. **本文档所定义的正式字段语义，优先于任何现有实现行为。** 若实现与本规范冲突，应判定为实现不符合规范，而非修改规范迁就实现。

---

## 2. 背景与问题背景

当前 XiaoMusic 前端存在以下已知问题，均由"后端未提供单一权威状态投影、前端不得不自行猜测播放真相"所导致：

- **未知歌曲问题**：切歌瞬间后端返回空标题，前端无法区分"真正的未知曲目"与"切歌过渡中的瞬态空值"，导致 UI 短暂显示"未知歌曲"。
- **进度条闪烁**：旧 revision 或延迟到达的响应覆盖了更新的状态，导致进度条回退闪烁。
- **选中项错位**：前端依赖 `cur_music` 文本在歌单中定位当前曲目，同名歌曲或列表序号变化时导致选中项错误对齐。
- **自动切歌不稳定**：前端无法可靠判断"切歌已完成"与"仍在切歌中"，导致切歌确认逻辑依赖超时猜测。
- **旧状态回写**：慢请求、乱序响应或页面恢复时，旧状态被写回 UI，覆盖更新的播放状态。

本规范通过定义统一状态快照模型，从根本上为上述问题提供解决基础。

---

## 3. 适用范围

本规范适用于：

- 后端：任何构建并对外暴露播放器状态的模块，包括但不限于 API 路由层、PlaybackCoordinator、PlaybackFacade。
- 前端：WebUI 中所有消费 `GET /api/v1/player/state` 或未来 SSE 流的代码路径。
- 测试与验收：所有涉及播放器状态一致性、切歌确认、进度同步的验收用例。

## 4. 设计原则

本规范遵循以下设计原则。所有实现与消费行为必须与这些原则保持一致。

### 4.1 单一权威状态来源

播放器的权威状态由服务端唯一决定。服务端的状态快照是唯一合法的"当前播放真相"，任何前端本地推算结果均不得凌驾于服务端快照之上。

### 4.2 可排序、可去乱序

状态快照必须携带单调递增的版本号（`revision`），使消费方能够在多路响应同时到达时，安全地丢弃过期状态，始终只应用最新状态。

### 4.3 可区分"切歌中"与"未知曲目"

"切歌中"是一个明确的传输状态（`transport_state`），不得通过清空 `track.title` 或返回空值来隐式表达。消费方必须能从 `transport_state` 字段直接判断当前是否处于切歌过渡中。

### 4.4 曲目身份的稳定标识

当前曲目必须有稳定的身份标识（`track.id`），不得仅靠 `track.title` 文本充当身份依据。`track.id` 在曲目切换时必须发生变化，在同一曲目的暂停/恢复过程中必须保持稳定。

### 4.5 支持前端平滑渲染，但不授权前端定义播放真相

前端可以在两次状态更新之间做短时视觉插值（例如进度条补帧），但这仅是展示层平滑处理，不得被视为对播放真相的重新定义。一旦收到更高 `revision` 的服务端状态，前端必须立即以服务端状态为准，不得保留本地插值结果继续渲染。

### 4.6 命令流与状态流解耦

播放控制命令（通过 `POST /api/v1/play`、`POST /api/v1/control/*` 等接口发出）与播放状态观测（通过 `GET /api/v1/player/state` 或 SSE 获取）是相互独立的通道。前端不得依赖命令响应体中的状态字段推断当前播放状态，正式播放状态只能通过状态查询通道获取。

## 5. 权威播放状态快照模型

本节定义权威播放状态快照（以下简称"状态快照"）的字段模型。所有对外状态输出必须与本节定义保持一致。

### 5.1 顶层结构

状态快照是一个 JSON 对象，包含以下字段分组：

- 顶层身份字段
- 播放传输状态
- 曲目对象（`track`）
- 上下文信息（`context`）
- 进度与时长

### 5.2 顶层身份字段

| 字段 | 类型 | 必须存在 | 说明 |
|---|---|---|---|
| `device_id` | `string` | 是 | 当前状态所属设备的唯一标识，非空字符串 |
| `revision` | `integer` | 是 | 状态版本号，单设备维度、单调递增，详见第 6 节 |
| `play_session_id` | `string` | 是 | 当前播放会话的唯一标识，详见第 7 节 |

`device_id` 必须与请求参数中的设备 ID 一致，不得返回其他设备的快照。

### 5.3 播放传输状态（`transport_state`）

| 字段 | 类型 | 必须存在 | 说明 |
|---|---|---|---|
| `transport_state` | `string` | 是 | 播放器当前传输状态，必须属于第 8 节定义的枚举值 |

`transport_state` 是整个状态快照中语义优先级最高的字段。消费方在解析状态快照时，必须首先读取 `transport_state`，再结合其他字段做展示决策。

### 5.4 曲目对象（`track`）

`track` 是描述当前曲目身份的对象。

| 字段 | 类型 | 必须存在 | 说明 |
|---|---|---|---|
| `track` | `object \| null` | 是 | 当前曲目信息；`transport_state` 为 `idle` 或 `stopped` 时可为 `null` |
| `track.id` | `string` | 是（若 `track` 非空） | 当前曲目的稳定身份标识，详见第 9 节 |
| `track.title` | `string` | 是（若 `track` 非空） | 当前曲目的展示标题 |

以下字段在 `track` 非空时可以包含，但消费方不得强依赖其存在：

| 字段 | 类型 | 说明 |
|---|---|---|
| `track.artist` | `string` | 曲目艺术家 |
| `track.album` | `string` | 曲目所属专辑 |
| `track.source` | `string` | 曲目来源标识（如 `local_library`、`jellyfin`、`site_media`、`direct_url`） |

重要约束：

- **`track.id` 是当前曲目的稳定身份标识。** UI 同步、去重、切歌判断必须优先依赖 `track.id`，而不是 `track.title`。
- `track.title` 可为空字符串，表示标题信息尚未解析完成，但这不意味着曲目身份未知。`track.id` 仍应在此时保持稳定。
- 当 `transport_state` 为 `switching` 或 `starting` 时，`track` 可以携带即将播放的曲目信息，也可以为 `null`，但不得通过 `track` 置空来隐式表达"切歌中"语义。切歌中的语义必须由 `transport_state` 承载。

### 5.5 上下文信息（`context`）

`context` 描述当前播放所属的上下文（例如歌单）及曲目在其中的位置。

| 字段 | 类型 | 必须存在 | 说明 |
|---|---|---|---|
| `context` | `object \| null` | 是 | 播放上下文；若无上下文则为 `null` |
| `context.id` | `string` | 是（若 `context` 非空） | 当前播放上下文的唯一标识（如歌单 ID） |
| `context.name` | `string` | 是（若 `context` 非空） | 当前播放上下文的展示名称 |
| `context.current_index` | `integer \| null` | 是（若 `context` 非空） | 当前曲目在上下文中的位置索引（从 0 开始），无法确定时为 `null` |

约束说明：

- `context.id` 与 `context.name` 分别承载上下文的身份与展示，消费方应使用 `context.id` 做逻辑判断，使用 `context.name` 做展示。
- `context.current_index` 仅表示当前曲目在当前上下文中的位置，不表示全局队列位置。消费方不得将其作为跨上下文的全局索引使用。
- `context` 为 `null` 表示当前播放不属于任何已知上下文（例如直接播放单个 URL），这是合法状态。消费方不得因 `context` 为 `null` 而推断播放出错。

### 5.6 进度与时长

| 字段 | 类型 | 必须存在 | 说明 |
|---|---|---|---|
| `position_ms` | `integer` | 是 | 当前播放位置，单位：毫秒；`transport_state` 为 `idle` 或 `stopped` 时应为 `0` |
| `duration_ms` | `integer` | 是 | 当前曲目总时长，单位：毫秒；未知时为 `0` |
| `snapshot_at_ms` | `integer` | 是 | 本次快照生成时的服务端时间戳，单位：毫秒（Unix 时间戳） |

约束说明：

- `position_ms` 是服务端在 `snapshot_at_ms` 时刻观测到的播放位置，是状态观测值，不是前端自行承诺的真值。
- 前端可以在两次快照之间，以 `snapshot_at_ms` 为基准、以本地时钟为参考做短时进度插值，使进度条展示更平滑。但一旦收到更高 `revision` 的新快照，必须立即以新快照的 `position_ms` 为准，放弃本地插值结果。
- 前端不得将本地插值结果写回或上报为"真实播放位置"。
- `duration_ms` 为 `0` 表示时长未知，不表示曲目时长为零或播放出错。

### 5.7 兼容字段定位

当前实现中存在以下旧字段，其正式语义如下：

| 旧字段 | 正式语义 | 兼容投影关系 |
|---|---|---|
| `cur_music` | 对应 `track.title` | 若旧字段继续出现，仅视为 `track.title` 的兼容投影输出，不再作为新实现的主依据 |
| `is_playing` | 对应 `transport_state == "playing"` | 若旧字段继续出现，仅视为 `transport_state` 的布尔兼容投影，不再作为新实现的主依据 |
| `offset` | 对应 `position_ms / 1000`（秒） | 若旧字段继续出现，仅视为 `position_ms` 的秒级兼容投影 |
| `duration` | 对应 `duration_ms / 1000`（秒） | 若旧字段继续出现，仅视为 `duration_ms` 的秒级兼容投影 |

以上旧字段如果继续存在于响应中，只能视为向后兼容的投影字段，不得作为新实现、新前端逻辑或新验收用例的主依据。新代码必须依赖本规范定义的正式字段。

## 6. revision 规范

### 6.1 定义

`revision` 是单设备维度、单调递增的状态版本号，类型为非负整数。

### 6.2 递增触发条件

任何会导致前端可见播放状态发生变化的事件，都必须触发 `revision` 前进。这包括但不限于：

- 播放新曲目（无论来源）
- 曲目自动切换（下一首、上一首）
- 暂停、恢复
- 停止
- 播放器进入或退出错误状态
- 曲目切换过程中的 `transport_state` 变化（如从 `playing` 变为 `switching`）

以下操作不触发 `revision` 前进：

- 进度位置的自然推进（前端通过插值处理，服务端不为每秒进度变化递增 revision）
- 不影响播放状态的内部操作（如日志记录、缓存刷新）

### 6.3 消费约束

- 前端只接受 `revision` 大于或等于当前已知 revision 的状态。
- 若收到的状态快照 `revision` 严格小于当前 UI 已渲染的 revision，该快照必须被丢弃，不得写回 UI。
- 多路并发请求（如同时发起的轮询请求）响应到达时，消费方必须以 `revision` 最大的那一份为准，其余丢弃。
- 页面刷新恢复时，前端不得用本地缓存状态覆盖服务端返回的状态，即便服务端状态的 `revision` 较小（例如服务端重启后 revision 重置），也应以服务端状态为准。

## 7. play_session_id 规范

### 7.1 定义

`play_session_id` 是标识一次连续播放会话的唯一字符串。其作用是让消费方能够明确区分"同一首歌的暂停恢复"与"切到了另一首歌"。

### 7.2 切换触发条件

以下情况必须生成新的 `play_session_id`：

- 切到不同曲目（无论是用户手动切换还是自动切歌）
- 从当前歌曲切到另一首同名歌曲
- 停止后重新播放任意曲目
- 通过 `POST /api/v1/play` 重新指定播放目标（即使目标与当前曲目相同，也视为新会话）

以下情况不得生成新的 `play_session_id`：

- 对当前曲目执行暂停
- 对当前暂停曲目执行恢复
- 仅调整音量

### 7.3 消费约束

前端必须将以下行为绑定到 `play_session_id` 边界：

- **展示标题的切换**：标题切换动画、歌名更新，必须以 `play_session_id` 变化为触发，而不是以 `cur_music` 字符串变化为触发。
- **进度条清零**：新会话开始时，进度条必须重置，不得延续上一会话的进度。
- **歌单选中项同步**：当前选中歌曲的高亮与定位，必须以 `play_session_id` + `track.id` 为依据，而不是以标题文本匹配为依据。
- **切歌确认**：前端等待切歌完成时，切歌成功的判据是收到 `play_session_id` 发生变化且 `transport_state` 变为 `playing` 的状态快照，而不是超时后猜测。

## 8. transport_state 规范

### 8.1 正式枚举值

`transport_state` 必须属于以下有限枚举集合：

| 枚举值 | 语义 |
|---|---|
| `idle` | 播放器未被使用，无曲目、无进度 |
| `starting` | 正在启动播放（首次解析、准备流、下发命令中），尚未确认设备侧开始播放 |
| `switching` | 正在从当前曲目切换到下一曲目，过渡期间 |
| `playing` | 设备侧已确认正在播放 |
| `paused` | 设备侧已确认处于暂停状态 |
| `stopped` | 设备侧已确认停止 |
| `error` | 播放器进入明确的异常状态，不可自动恢复 |

### 8.2 各状态语义约束

**`idle`**：
- 播放器未被激活，`track` 应为 `null`，`position_ms` 为 `0`。
- 与 `stopped` 的区别：`stopped` 表示曾经在播放但已明确停止；`idle` 表示从未开始或已完全归零。

**`starting`**：
- 播放命令已下发，但尚未从设备侧得到确认播放开始的反馈。
- `track` 可以携带即将播放的曲目信息（如已解析到），也可以为 `null`。
- 不得通过清空 `track.title` 来表达"正在准备"的语义；应直接使用 `starting` 状态。

**`switching`**：
- 正在从一首曲目切换到另一首，处于两首之间的过渡期。
- 不得通过 `track` 置空或返回空标题来表达"切歌中"。切歌中的语义必须由 `switching` 承载。
- 消费方收到 `switching` 状态时，应保留上一首曲目的展示状态（如标题、专辑封面），而不是清空显示。

**`playing`**：
- 设备侧已确认当前正在播放，`track` 必须非空，`track.id` 必须稳定。
- `position_ms` 反映观测到的播放位置。

**`paused`**：
- 设备侧已确认当前处于暂停状态，`track` 必须非空，`track.id` 必须与暂停前保持一致。
- `position_ms` 应保持在暂停时刻的观测值，不再推进。

**`stopped`**：
- 播放已明确停止。`track` 可以保留最后播放曲目信息，也可以为 `null`，具体由实现决定并保持一致。
- `position_ms` 应为 `0` 或暂不定义。

**`error`**：
- 播放器进入明确异常状态（例如解析失败、设备不可达且已超过重试上限）。
- `error` 状态不用于表达"数据暂时缺失"或"切歌中间态"。这两种情况分别由 `track.title` 为空与 `switching` 状态承载。
- 进入 `error` 后，消费方应向用户呈现明确的错误提示，而不是继续轮询等待状态自动恢复。

### 8.3 禁止用隐式方式表达传输状态

以下隐式表达方式不得再使用：

- 通过 `track.title` 为空表达"切歌中"
- 通过 `is_playing: false` 且 `cur_music` 变化表达"正在切歌"
- 通过 `offset` 为 `0` 且 `duration` 不变表达"正在启动"

所有传输状态的语义必须由 `transport_state` 字段显式承载。

## 9. track identity 规范

### 9.1 track.id 的定义

`track.id` 是当前曲目在 XiaoMusic 系统内的稳定身份标识。其定义如下：

- 同一首曲目在同一次连续播放会话内，`track.id` 必须保持不变。
- 切换到不同曲目后，`track.id` 必须发生变化。
- 同名歌曲（`track.title` 相同）出现在不同位置时，`track.id` 必须能够区分它们，不得相同。

### 9.2 track.id 的生成规则

当上游来源（本地库、Jellyfin 等）提供天然稳定 ID 时，应直接使用上游 ID 作为 `track.id`。

当上游来源不提供天然稳定 ID 时，系统必须生成稳定的投影 ID。投影 ID 必须满足：

- 基于 `context.id`（若存在）+ `context.current_index`（若存在）+ `track.title` 的组合生成。
- 采用确定性哈希算法，保证相同输入始终产生相同输出。
- 若 `context` 为 `null`（直接播放单 URL），则基于 `track.title` + `stream_url` 的组合生成。

禁止将 `track.title` 直接用作 `track.id`，即便在没有上下文的情况下也不允许。标题不具备唯一性，不得充当身份标识。

### 9.3 消费约束

- 前端的歌曲选中同步、切歌确认、进度条绑定，必须优先依赖 `track.id`，而不是 `track.title`。
- 歌单列表的高亮选中逻辑，必须通过 `track.id` 与列表项 ID 对照完成，不得依赖 `cur_music` 字符串与列表项名称的文本匹配。
- 若 `track.id` 未发生变化但 `track.title` 发生了变化（例如标题延迟到达），消费方应更新展示标题，但不应将其视为曲目切换事件。

## 10. 上下文与进度字段规范

### 10.1 上下文字段语义

`context.id` 是播放上下文的稳定标识，用于：

- 判断播放上下文是否发生切换（例如从歌单 A 切到歌单 B）。
- 结合 `context.current_index` 定位当前曲目在上下文中的位置。

`context.name` 仅用于展示，不得用于逻辑判断。

`context.current_index` 反映当前曲目在上下文中的位置索引（从 0 开始）。消费方可将其用于：

- 歌单列表当前播放项的高亮定位（配合 `track.id`）。
- 判断当前是否有上下文内的下一首或上一首。

`context.current_index` 为 `null` 时，表示当前曲目的上下文位置无法确定（例如随机模式下未跟踪索引）。消费方不得将 `null` 视为错误，应将其作为合法状态处理。

### 10.2 进度字段精度与单位

`position_ms` 与 `duration_ms` 的单位为毫秒。

当前实现中 `offset` 与 `duration` 的单位为秒。兼容过渡期内，如果旧字段继续存在，消费方必须注意单位差异，不得混用。新实现一律以毫秒字段为准。

### 10.3 进度插值边界

前端允许在 `snapshot_at_ms` 基础上，使用本地时钟对 `position_ms` 做线性插值，使进度条展示更平滑。插值规则如下：

- 仅在 `transport_state == "playing"` 时允许做正向插值。
- `transport_state` 为 `paused`、`switching`、`starting`、`stopped`、`idle`、`error` 时，不得推进进度。
- 插值时长不得超过两次状态快照之间的合理间隔（建议不超过 5 秒）；超出后应停止推进，等待下一次快照。
- 收到新快照后，无论插值结果如何，必须立即以新快照的 `position_ms` 为准。

## 11. 统一状态投影约束

本节定义"统一状态快照构建器"的硬约束。这是本规范最核心的实现约束。

### 11.1 唯一快照构建器

系统必须存在且只存在一个状态快照构建器（State Projection Builder），负责生成符合本规范的状态快照对象。

以下各输出路径必须共享同一个快照构建器：

- `GET /api/v1/player/state` 的 `data` 字段内容
- 未来 `GET /api/v1/player/stream` SSE 事件的 `data` 字段内容
- 任何其他对外暴露播放器状态的输出路径

### 11.2 禁止各路径独立拼装状态

以下行为不得存在：

- HTTP 查询接口自行从设备对象读取字段拼装状态
- SSE 推送路径另行订阅设备事件并独立拼装状态
- WebUI 前端通过本地逻辑"补全"后端未返回的字段，并将补全结果作为播放真相

### 11.3 快照构建器的职责边界

状态快照构建器的职责包括：

- 从 PlaybackCoordinator 或等效的权威内部状态获取原始播放事实
- 生成或查找 `revision`、`play_session_id`、`track.id` 等身份字段
- 将内部状态映射到本规范定义的 `transport_state` 枚举
- 输出符合本规范字段模型的状态快照对象

状态快照构建器不得直接从设备驱动层读取原始值绕过 PlaybackCoordinator，也不得在输出路径（API 路由层或 SSE 推送层）内嵌入状态拼装逻辑。

## 12. 前端消费边界

### 12.1 前端是状态消费者，不是状态裁决者

前端的职责是忠实呈现服务端状态快照，而不是对状态做重新裁决或补充定义。以下行为明确禁止：

- 前端自行维护一套"本地认定的当前歌曲"，与服务端快照并列，并在冲突时选择本地版本。
- 前端通过 `rememberedPlayingSong`、`pendingTitle`、`boundaryGuess` 等本地机制覆盖服务端返回的 `track.id` 或 `track.title`。
- 前端认为服务端状态"不够准确"，并以本地轮询结果替代服务端状态作为展示依据。

### 12.2 本地容错机制的合法边界

前端可以维护以下有限的本地容错机制：

- **短时进度插值**：在两次快照之间，对 `position_ms` 做线性插值以平滑进度条，但插值结果不得写回或上报为真实状态（见第 10.3 节）。
- **页面刷新过渡展示**：在页面刷新后、首次快照到达前，可短暂展示上次已知状态，用于避免白屏。此过渡期不得超过第一次有效快照到达后延续，收到快照后必须立即以快照为准。
- **切歌动画缓冲**：在收到 `switching` 状态时，可在 UI 上保持上一首曲目的展示信息，直到收到 `play_session_id` 变化且 `transport_state` 为 `playing` 的新快照。

以上容错机制仅允许在服务端状态尚未到达的空窗期临时生效，收到有效快照后必须立即让位。

### 12.3 状态同步的优先依据

前端进行以下操作时，必须以指定字段为优先依据：

| 操作 | 优先依据 |
|---|---|
| 判断当前播放的是哪首曲目 | `play_session_id` + `track.id` |
| 高亮歌单中当前播放项 | `track.id` 与列表项 ID 的对照 |
| 判断切歌是否已完成 | `play_session_id` 变化且 `transport_state == "playing"` |
| 更新进度条初始值 | `position_ms`（以 `snapshot_at_ms` 为观测基准） |
| 判断是否正在切歌 | `transport_state == "switching"` |
| 判断是否处于暂停 | `transport_state == "paused"` |
| 判断播放器是否出错 | `transport_state == "error"` |
| 丢弃过期状态 | `revision` 比较 |

## 13. 错误与缺省处理原则

### 13.1 数据暂缺、切歌中、真正异常，三者必须明确区分

| 情况 | 正确表达方式 | 禁止的表达方式 |
|---|---|---|
| 标题信息尚未解析完成 | `track.title` 为空字符串，`track.id` 保持稳定，`transport_state` 为 `playing` 或 `starting` | 将 `transport_state` 设为 `error`；消费方将其展示为"未知歌曲" |
| 正在切换到下一首 | `transport_state == "switching"` | 清空 `track`；将 `is_playing` 设为 `false`；通过 `cur_music` 为空表达 |
| 播放器进入真正异常 | `transport_state == "error"` | 无 |

### 13.2 metadata 未完全解析时的处理

当曲目 metadata（如标题、时长）尚未完全解析时：

- `track.id` 必须保持稳定，不得因 metadata 缺失而生成新的 `play_session_id`。
- `track.title` 可以为空字符串，但不得为 `null`（若 `track` 对象本身非空）。
- `duration_ms` 未知时应为 `0`，消费方收到 `duration_ms == 0` 时不得将其理解为"播放出错"。
- 消费方不得因为 `track.title` 为空而回退显示上一首曲目的标题，或跳到下一首。

### 13.3 禁止因中间态拼装导致的状态回退

以下行为明确禁止：

- 因 `transport_state` 处于 `switching` 或 `starting` 而导致 UI 回退到显示上一首（除非这是设计明确要求的过渡动效）。
- 因服务端快照某字段暂缺而触发前端"推测下一首"逻辑，主动将 UI 切换到顺序下一首。
- 在 `revision` 未变化的情况下，因多次重复轮询收到同一快照而触发不必要的 UI 重渲染。

### 13.4 旧状态不得回写

前端在以下情况下必须主动丢弃旧状态，不得将其回写 UI：

- 收到的快照 `revision` 小于当前已渲染状态的 `revision`。
- 页面可见性恢复（从后台切回前台）后，收到服务端快照前不得将 localStorage / sessionStorage 中的旧状态写入展示层。

## 14. 与其他规范的关系

本节说明本文档在整体文档体系中的定位，以及与相关规范的上下位关系。

### 14.1 本文档是上位状态语义规范

本文档定义"权威播放状态快照"的结构、字段语义与消费约束，是以下所有文档的共同上位规范。当下位文档与本文档存在冲突时，以本文档为准。

### 14.2 与 `docs/api/api_v1_spec.md` 的关系

`docs/api/api_v1_spec.md` 是 XiaoMusic Runtime v1 API 的唯一权威契约来源，承接本文档的状态语义，定义 `GET /api/v1/player/state` 的具体 HTTP 接口契约，包括请求参数、响应 envelope 结构、错误模型与兼容字段处理。

本文档与 `api_v1_spec.md` 的分工：本文档定义"快照应该包含什么语义"，`api_v1_spec.md` 定义"快照如何通过 HTTP 对外暴露"。

### 14.3 与 `docs/spec/player_stream_sse_spec.md` 的关系

当 `docs/spec/player_stream_sse_spec.md` 建立后，该文档承接本文档的状态语义，定义 SSE 事件的传输协议、连接管理、心跳机制与断线重连约束。SSE 事件中的状态数据字段，必须与本文档定义的状态快照模型保持完全一致。

### 14.4 与 WebUI 相关规范的关系

`docs/spec/webui_playback_state_machine_spec.md` 与 `docs/spec/webui_playback_state_machine_mapping.md` 定义 WebUI 的前端展示状态机与实现映射。这些文档定义前端如何消费本文档所描述的状态快照，但不得在前端侧重新定义播放真相的来源。前端规范中凡涉及"以哪个字段为权威依据"的判断，必须与本文档第 12.3 节保持一致。

### 14.5 与 `docs/spec/playback_coordinator_interface.md` 的关系

`docs/spec/playback_coordinator_interface.md` 定义 PlaybackCoordinator 的内部接口契约、Adapter 职责与 Framework 职责。PlaybackCoordinator 是状态快照构建器的内部状态来源；其对外暴露的播放事实（`current_index`、`context`、`track` 等）必须通过本文档定义的统一快照构建器转化为对外输出，不得直接绕过快照构建器暴露原始内部状态。

### 14.6 与 `docs/spec/runtime_specification.md` 的关系

`docs/spec/runtime_specification.md` 定义 `xiaomusic/core/` 的数据模型、接口契约与错误体系。其中 `ResolvedMedia`、`TransportDispatchResult` 等内部数据模型是构建状态快照的原始输入来源，但其字段不得直接透传为对外状态快照字段；必须经过状态快照构建器的规范化映射。
